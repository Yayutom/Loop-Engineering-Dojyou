プロンプトを書く人を、
ループの設計者へ。

2026年、AIの腕前は「プロンプトの言い回し」では決まらなくなりました。決め手はループ設計（Loop Engineering）——エージェントが何をきっかけに動き、どう繰り返し、何を「合格」とし、いつ止まるか。読むだけでなく、シミュレータと演習で手を動かしながら身につけます。

01なぜ今「ループ設計」なのか

あなたは今まで、AIに1回お願いして1回返事をもらう使い方をしてきたはずです。これがプロンプトエンジニアリング。一方エージェントは、自分で動いて結果を見て、また動く——この繰り返し（ループ）で目標に到達します。

これまで / Prompting

1往復で終わる

人間：「このバグ直して」→ AI：「こうですよ」→ 人間が確認 → また指示…。毎回あなたが回している。

これから / Loop Engineering

AIが自分で回す

「テストが全部通るまで直し続けて」とループの仕組みを渡す → AIが直す→テスト実行→失敗を見て直す…を自動で繰り返す。

2026年6月の出来事。 Addy Osmani が "Loop Engineering" を提唱し、Peter Steinberger が一文に要約しました——「エージェントにプロンプトするのをやめ、エージェントをプロンプトする"ループ"を設計せよ」。プロンプトの言い回しがボトルネックでなくなった今、価値は「トリガー・トポロジー・検証・停止条件」の設計に移りました。

この道場で身につくこと

  • ループを構成する4つの部品を、自分の言葉で説明できる
  • シミュレータで「検証なしループ」が静かに失敗する様子を体感する
  • 擬似コードで自分のループを1本書けるようになる
  • 明日から使える実務チェックリストを持ち帰る
✓ 完了！

初心者がつまずく所： 「観測（OBSERVE）」と「検証（VERIFY）」は別物です。観測は結果を受け取ること。検証はその結果が"本当に合格か"を判定すること。テストの出力を読むのが観測、テストが全部緑かを判定するのが検証。検証を省くとループは"できたつもり"で止まります。

034つの設計要素

ループ設計でいじるツマミは、突き詰めると4つだけ。カードをクリックして展開しましょう。これがこの道場の背骨です。

① Trigger（トリガー / きっかけ）
いつループを起動するか
▸

ループは誰が始める？ 自分で毎回叩くなら、まだ自動化ではありません。トリガーの例：

  • 手動：人がコマンドを打つ（学習の入口）
  • スケジュール：毎朝9時、5分ごと（cron 的）
  • イベント：PRが来たら / Slackにメンションが来たら / CIが落ちたら

初心者の最初の一歩は「手動トリガー」で十分。慣れたらイベント駆動へ。

② Topology（トポロジー / 形）
ループの構造・誰が何回まわるか
▸

ループの"かたち"。単純な1本ループから、役割分担まで。

  • 単一ループ：1体のエージェントが回す（まずこれ）
  • 生成→検証の二人羽織：作る係と、ダメ出しする係を分ける
  • マルチエージェント：調査・実装・レビューを別エージェントに
  • 入れ子ループ：大きな目標ループの中に小さな修正ループ

迷ったら小さく。トポロジーを複雑にするほどデバッグは難しくなります。

③ Verifier（検証器） 最重要
「合格 / 完了」を機械的に判定する仕組み
▸

ループの品質はここで決まります。「いい感じ」「たぶんOK」はNG。機械が○×を返せる基準にするのがコツ。

  • テストが全部パスする（緑になる）
  • Lint / 型チェックがエラー0
  • ビルドが通る / 指定の文字列が出力に含まれる
  • 別のLLMが「要件を満たす」と判定（LLM-as-judge）

2026年の合言葉：「ボトルネックはモデルではなく検証器」。"良い"と"完了"を定義する力が、いま一番の希少スキルです。

④ Stop rules（停止条件）
いつ抜ける / いつ諦めるか
▸

止め方を決めないループは、お金（トークン）を燃やし続けます。最低でも2種類の停止口を：

  • 成功で停止：検証器が合格 → 即終了（本命）
  • 上限で停止：最大N回 / トークン上限 / 時間切れ（暴走防止の安全網）
  • 停滞で停止：M回連続で進歩なし → 諦めて人間にエスカレーション

「成功」だけだと永遠に終わらないリスク、「上限」だけだと未完成で打ち切るリスク。両方置くのが定石。

暗記用の一文： 「トリガーで始まり、トポロジーという形で回り、検証器が合否を決め、停止条件で抜ける。」
✓ 完了！

必ず試してほしい3パターン：
  1. 検証器「なし」で実行 → エージェントは1回で「できました！」と止まるのに、テストはまだ失敗。これが"できたつもり"の偽完了。
  2. 検証器「テスト」+ 停止「合格で停止」 → 失敗0になった瞬間に賢く止まる。これが理想形。
  3. 停止「上限まで回す」+ 最大15回 → 合格後もムダに回り続け、トークンを浪費。停止条件の大切さがわかります。

05ハンズオン② Verifier道場

「検証器がボトルネック」を体で覚える回。下の各タスクで、使える検証器（機械が○×を返せるもの）を選んでください。曖昧なものを選ぶと、ループは止まれません。

良い検証器の見分け方： ①自動で実行できる ②結果が○か×かはっきりする ③人間の気分で揺れない。この3つを満たさないものは「観測」止まりで、停止条件には使えません。
✓ 完了！

06ループを正しい順に組み立てる

下のステップは順番がバラバラです。ドラッグして、正しいループの1周になるよう並べ替え、採点を押してください。

  • ⠿取得した結果（テスト出力など）を読む【観測】
  • ⠿タスクと現状から次の一手を決める【計画】
  • ⠿合格 or 上限なら抜ける、ダメなら最初へ戻る【停止判定】
  • ⠿ツール（編集・コマンド実行）で実際に手を動かす【行動】
  • ⠿結果が"合格基準"を満たすか機械的に判定する【検証】
✓ 完了！

  • ✓まず小さく回したか？（単一ループ→慣れたら多段/マルチへ）
  • ✓取り返しのつかない操作に承認ゲートを置いたか？（本番DB削除・送金・公開など）

よくある失敗とその処方

症状原因処方
1周で「できました」と止まるが成果が嘘検証器がない／弱い機械判定できる合格条件を追加（テスト・Lint・スキーマ）
永遠に終わらない／トークンが溶ける停止条件が「成功」だけ最大回数・停滞検知・コスト上限を追加
途中から的外れな修正を始めるコンテキストが汚れて肥大毎周コンテキストを剪定、要点だけ渡す（文脈設計）
同じ失敗を繰り返す失敗から学ぶ仕組みがない検証の不合格理由を次の計画に必ず渡す
たまに危険な操作をする承認ゲート不在不可逆操作の前に人間 or ルールで確認
コンテキスト設計（おまけの最重要トピック）： 2025年のChromaの研究で、情報を増やすほどモデルは賢くならず、むしろ精度が落ちることが18モデルで確認されました。ループの各周で「全部盛り」を渡さず、今この一手に必要な要点だけを渡す——これがLoop Engineeringと両輪のContext Engineeringです。

A用語集（迷ったらここへ）

Loop Engineering

エージェントが自分で繰り返し動く"ループ"そのものを設計する実践。プロンプト調整の次の段階。

Agent（エージェント）

LLM＋ツール＋ループ。指示を1回返すだけでなく、行動→観測→再行動を自律的に繰り返す。

Trigger

ループの起動きっかけ。手動／スケジュール／イベント。

Topology

ループの構造。単一／生成＋検証／マルチエージェント／入れ子。

Verifier（検証器）

合格・完了を機械的に判定する仕組み。ループ品質の心臓部。

Stop rule（停止条件）

いつ抜けるか／諦めるか。成功・上限・停滞の組み合わせ。

ReAct

Reason+Act。「考える→動く→観測」を繰り返す古典的ループ。Loop Engineeringの土台。

Context Engineering

各周でモデルに"何を見せるか"を厳選する技術。多く渡すほど良いとは限らない。

LLM-as-judge

別のLLMに合否を判定させる検証手法。自動テストが書けない曖昧タスク向け。

Harness（ハーネス）

ループ・ツール・停止条件をまとめてエージェントに被せる"足回り"全体。
